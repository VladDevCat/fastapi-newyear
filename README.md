Holiday Prep API - это REST API для веб-приложения по подготовке к праздникам.  
Проект реализован на **Python 3.11 + FastAPI** и является продолжением лабораторной работы №1.

В проекте реализованы:

- REST API для работы с сущностью `HolidayItem`
- PostgreSQL 18
- ORM через SQLAlchemy
- миграции через Alembic
- мягкое удаление (Soft Delete)
- пагинация для списковых запросов
- модульная архитектура
- запуск через Docker и Docker Compose

Сущность `HolidayItem` описывает элемент подготовки к празднику и содержит:

- `id` - UUID
- `title` - название
- `description` - описание
- `status` - статус (`planned`, `purchased`, `done`)
- `created_at` - дата создания
- `updated_at` - дата обновления
- `deleted_at` - дата мягкого удаления

- Получить список активных элементов

GET /items

Параметры пагинации передаются через query string:
page номер страницы, по умолчанию 1
limit количество элементов на странице, по умолчанию 10
limit должен быть в диапазоне от 1 до 100

Пример запроса:
curl "http://localhost:4200/items?page=1&limit=10"

Пример ответа:

{
  "meta": {
    "page": 1,
    "limit": 10,
    "total_items": 0,
    "total_pages": 0
  },
  "items": []
}

- Получить активный элемент по ID
GET /items/{id}
Пример:
curl http://localhost:4200/items/ID_ЭЛЕМЕНТА

- Создать новый элемент

POST /items

Пример запроса:

curl -X POST http://localhost:4200/items \
  -H "Content-Type: application/json" \
  -d "{\"title\":\"Купить гирлянду\",\"description\":\"Нужна гирлянда для ёлки\",\"status\":\"planned\"}"

Пример тела запроса:

{
  "title": "Купить гирлянду",
  "description": "Нужна гирлянда для ёлки",
  "status": "planned"
}

Статус успеха: 201 Created

- Частично обновить элемент

PATCH /items/{id}

Пример:

curl -X PATCH http://localhost:4200/items/ID_ЭЛЕМЕНТА \
  -H "Content-Type: application/json" \
  -d "{\"status\":\"done\"}"

Статус успеха: 200 OK

---
HTTP-статусы
В проекте используются следующие коды ответа:

- 200 OK успешное получение или обновление данных
- 201 Created успешное создание записи
- 204 No Content успешное мягкое удаление
- 400 Bad Request некорректные входные данные
- 404 Not Found ресурс не найден или уже помечен как удалённый
- 409 Conflict конфликт данных
- 422 Validation Error
- 500 Internal Server Error внутренняя ошибка сервера

Все 6 HTTP-эндпоинтов работают корректно
В приложении реализованы:
- GET /items
- GET /items/{id}
- POST /items
- PUT /items/{id}
- PATCH /items/{id}
- DELETE /items/{id}

Реализована пагинация

Для эндпоинта GET /items используются параметры:
- page
- limit
Ответ включает:
- текущую страницу
- размер страницы
- общее количество элементов
- общее количество страниц
При запросах с разными значениями page возвращаются разные части списка.

Валидация данных

В проекте используется валидация DTO через Pydantic.
Проверяются:
- обязательность полей
- длина строк
- допустимые значения статуса
- корректность параметров пагинации
- невозможность отправить пустой PATCH-запрос

Использование переменных окружения

Для конфигурации используются переменные окружения из файла .env.
Через них задаются:
- пользователь базы данных
- пароль базы данных
- имя базы данных
- хост базы данных
- порт базы данных
Это позволяет не хардкодить чувствительные настройки в исходном коде.

Миграции базы данных

Схема базы данных создаётся и изменяется только через миграции Alembic.
В репозитории присутствуют файлы миграций, которые позволяют:
развернуть структуру базы данных с нуля
воспроизводимо создать нужную схему на чистом PostgreSQL
Ручное создание таблиц через SQL-клиенты не используется.

Архитектура приложения

Проект построен по модульной архитектуре с разделением ответственности.

- Контроллер (api)
Отвечает за:
обработку HTTP-запросов
вызов сервисного слоя
возврат HTTP-ответов и статус-кодов

- Сервис (services)
Содержит:
бизнес-логику приложения
правила создания, обновления, удаления и поиска данных

- Репозиторий (repositories)
Отвечает за:
работу с ORM
выполнение запросов к базе данных

- Модель (models)
Содержит:
ORM-сущности SQLAlchemy
описание таблиц и полей

- DTO / схемы (schemas)
Отвечают за:
валидацию входных данных
описание структуры ответов API
